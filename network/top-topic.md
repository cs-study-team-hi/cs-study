1. TCP와 UDP의 차이점에 대해서 설명해보세요.
초기 답변)
우선 둘다 5계층 기준으로 4번째 레이어인 트랜스포트 레이어로 애플리케이션을 지나가기 전 단계인 프로토콜입니다.

이 때 tcp와 같은 경우, 혼잡 제어, 에러 제어, 흐름 제어 등을 보장하기 위해 여러 규칙을 넣었지만
udp와 같은 경우는 저런 problem에 대한 완전한 보장을 프로토콜 자체만으로 하지 않음

tcp는 세그먼트가 보장을 하려다 보니 두꺼움
udp는 상대적으로 얇아졌다.

데이터 손실의 영향이 강력한 상황에서는 tcp를 사용하고, 동영상 시청과 같이 데이터의 손실이 약간 허용되면 udp를 사용하면 됩니다.

> 깔끔한 답변
> TCP와 UDP는 모두 전송 계층 프로토콜이고, 핵심 차이는 신뢰성과 속도의 트레이드오프입니다.
TCP는 연결 지향이며 순서 보장, 재전송, 흐름 제어, 혼잡 제어를 통해 신뢰성 있는 전송을 제공합니다.
반면 UDP는 비연결형이라 별도의 보장 기능이 적어서 더 가볍고 빠릅니다.
그래서 정확한 전달이 중요하면 TCP, 지연 최소화가 중요하면 UDP를 사용합니다.
> 

2. GET과 POST의 차이점에 대해서 설명해보세요.
> 우수 답변
> GET과 POST는 둘 다 HTTP 요청 방식이라는 공통점이 있습니다.

다만 차이는 크게 세 가지가 있습니다.

첫째, 목적 측면에서  
GET은 서버에서 데이터를 조회할 때 주로 사용되고,  
POST는 서버에 새로운 데이터를 전달하거나 생성할 때 주로 사용됩니다.

둘째, 데이터 전달 방식 측면에서  
GET은 보통 데이터를 URL의 쿼리 파라미터에 담아 보내고,  
POST는 보통 데이터를 HTTP Body에 담아 보냅니다.

셋째, 특성 측면에서  
GET은 같은 요청을 여러 번 보내도 서버 상태를 바꾸지 않는 조회에 적합하고,  
POST는 요청에 따라 서버 상태가 바뀔 수 있어서 데이터 등록이나 처리에 적합합니다.

그래서 단순 조회가 중요하면 GET을,  
데이터 생성이나 변경 처리가 중요하면 POST를 선택합니다.

3. HTTP와 HTTPS의 차이점에 대해서 설명해보세요.

둘 다 웹에서 클라이언트와 서버가 통신하기 위한 프로토콜이라는 공통점이 있습니다.

다만 차이는 크게 세 가지가 있습니다. 첫째, 보안 측면에서 HTTP는 데이터를 평문으로 주고받기 때문에 중간에서 내용을 볼 수 있고 변조될 위험도 있지만, HTTPS는 SSL/TLS를 통해 데이터를 암호화해서 전송하므로 도청이나 변조를 방지할 수 있습니다.
둘째, 동작 방식 측면에서 HTTP는 요청과 응답을 바로 주고받는 반면, HTTPS는 통신 전에 TLS 핸드셰이크를 통해 서버 인증서를 검증하고 암호화에 사용할 세션을 만든 뒤 데이터를 주고받습니다.
셋째, 성능과 사용 상황 측면에서 HTTP는 구조가 단순해서 약간 가볍지만, HTTPS는 현대 환경에서는 성능 부담이 많이 줄었고 보안이 중요한 운영 서비스에서는 사실상 기본입니다.
그래서 단순한 내부 테스트나 민감하지 않은 통신이면 HTTP를 사용할 수 있지만, 로그인, 결제, 개인정보 처리처럼 보안과 신뢰가 중요한 상황에서는 HTTPS를 선택하는 것이 맞습니다.

RESTful은 REST 원칙을 잘 따르도록 설계한 API 방식입니다.
이 개념은 서버와 클라이언트가 자원을 일관되고 직관적으로 주고받기 위해 필요합니다.
핵심 특징은 크게 세 가지입니다.

첫째, 자원을 URI로 표현합니다.
즉 /members/1, /orders/10처럼 행위가 아니라 자원 중심으로 주소를 설계합니다.

둘째, 행위는 HTTP 메서드로 구분합니다.
같은 자원이라도 조회는 GET, 생성은 POST, 수정은 PUT 또는 PATCH, 삭제는 DELETE처럼 표현합니다.

셋째, 무상태성(stateless)을 지향합니다.
즉 각 요청은 서로 독립적이며, 서버는 이전 요청의 상태를 계속 기억하지 않는 방향으로 설계합니다.

예를 들어 회원 정보를 조회할 때는 GET /members/1, 주문을 생성할 때는 POST /orders처럼 설계할 수 있습니다.

그래서 RESTful은 자원을 중심으로 API를 설계하고, HTTP의 의미를 자연스럽게 살려서 확장성과 일관성을 높이는 방식이라고 볼 수 있습니다.

<summary>3. TCP의 3-way handshake와 4-way handshake에 대해 설명해주세요.</summary>

TCP는 신뢰성 있는 연결 지향 프로토콜로,  
데이터를 주고받기 전에는 연결을 맺고, 통신이 끝나면 연결을 종료하는 절차를 거칩니다.

먼저 3-way handshake는 연결 수립 과정입니다.  
클라이언트가 SYN을 보내 연결을 요청하고,  
서버는 SYN+ACK로 이를 수락하며,  
마지막으로 클라이언트가 ACK를 보내면 연결이 성립됩니다.

이 과정의 목적은  
서로 송수신이 가능한 상태인지 확인하고,  
초기 시퀀스 번호를 동기화해서 신뢰성 있는 통신을 준비하는 것입니다.

다음으로 4-way handshake는 연결 종료 과정입니다.  
클라이언트가 FIN을 보내면 서버는 ACK로 응답하고,  
남아 있는 데이터를 모두 전송한 뒤 서버가 별도로 FIN을 보냅니다.  
그 후 클라이언트가 ACK를 보내면 연결이 종료됩니다.

종료가 3-way가 아니라 4-way인 이유는  
TCP가 Half-Close를 지원하기 때문입니다.  
즉, 한쪽이 먼저 전송을 끝내더라도 다른 쪽은 아직 데이터를 더 보낼 수 있어서  
ACK와 FIN을 분리해서 처리합니다.

추가로 클라이언트는 마지막 ACK를 보낸 뒤 바로 닫지 않고  
TIME_WAIT 상태로 잠시 대기합니다.  
이것은 마지막 ACK가 유실되었을 때 서버가 FIN을 다시 보낼 수 있게 하고,  
이전 연결의 지연 패킷이 다음 연결과 섞이는 것을 방지하기 위한 장치입니다.

면접에서 45초 정도로 짧게 말하면  
“TCP는 연결 지향 프로토콜이라 연결 수립에는 3-way handshake, 종료에는 4-way handshake를 사용합니다.  
3-way는 SYN → SYN+ACK → ACK 순서로 연결을 맺고,  
4-way는 FIN → ACK → FIN → ACK 순서로 연결을 종료합니다.  
종료가 4단계인 이유는 Half-Close 때문에 한쪽이 종료 요청을 보내도 다른 쪽은 남은 데이터를 더 보낼 수 있기 때문입니다.”  
정도로 답할 수 있습니다.

1분 정도로 말하면  
시퀀스 번호 동기화, CLOSE_WAIT, TIME_WAIT까지 함께 언급하면 좋습니다.  
예를 들어  
“3-way handshake는 서로의 송수신 가능 여부와 초기 시퀀스 번호를 확인하는 과정입니다.  
반면 4-way handshake는 연결 종료 과정인데, 클라이언트가 FIN을 보내면 서버는 ACK로 받고,  
남은 데이터를 정리한 뒤 FIN을 보내며, 클라이언트가 마지막 ACK를 보내고 TIME_WAIT 후 종료합니다.  
특히 종료가 4-way인 이유는 양방향 종료가 독립적이기 때문입니다.”  
이렇게 말하면 조금 더 깊이 있어 보입니다.

1분 30초 정도로 자세히 말하면  
“TCP는 신뢰성 있는 연결 지향 프로토콜이라 데이터 전송 전후에 연결 수립과 종료 절차를 거칩니다.  
먼저 3-way handshake에서는 클라이언트가 SYN을 보내 연결을 요청하고,  
서버가 자신의 초기 시퀀스 번호와 ACK 번호를 담은 SYN+ACK로 응답한 뒤,  
클라이언트가 최종 ACK를 보내면서 연결이 성립됩니다.  
이 과정은 단순히 연결만 여는 것이 아니라, 서로 송수신이 가능한지 확인하고  
초기 시퀀스 번호를 동기화해서 신뢰성 있는 통신을 준비하는 과정입니다.  
여기서 시퀀스 번호를 임의의 값으로 시작하는 이유는 이전 연결에서 지연된 패킷과 현재 연결을 구분하기 위해서입니다.

연결 종료 시에는 4-way handshake를 사용합니다.  
클라이언트가 FIN을 보내면 서버는 ACK로 응답하고 CLOSE_WAIT 상태로 들어갑니다.  
이때 서버는 아직 보낼 데이터가 남아 있을 수 있기 때문에 바로 연결을 끊지 않고,  
남은 데이터를 모두 전송한 뒤 별도로 FIN을 보냅니다.  
그 후 클라이언트가 ACK를 보내면 서버는 연결을 닫고,  
클라이언트는 바로 종료하지 않고 TIME_WAIT 상태로 잠시 대기한 뒤 완전히 종료합니다.

종료가 4단계인 이유는 TCP가 Half-Close를 지원하기 때문입니다.  
즉, 양방향 연결이 독립적으로 닫히기 때문에 한쪽이 종료 의사를 보냈다고 해서  
다른 쪽의 전송까지 즉시 끝나는 것은 아닙니다.  
또한 TIME_WAIT는 마지막 ACK가 유실되었을 때 서버가 FIN을 재전송할 수 있도록 하고,  
이전 연결의 지연 패킷이 다음 연결과 섞이는 문제를 방지합니다.  
실무에서는 TIME_WAIT 소켓이 많이 쌓이면 포트 자원 문제가 생길 수 있어서  
Keep-Alive나 Connection Pool을 활용해 재연결 비용을 줄이는 방식도 자주 사용합니다.”  
이렇게 답하면 됩니다.

<summary>Q. HTTPS와 SSL Handshake에 대해 설명해보세요.</summary>

---

### ⏱️ 45초 버전

HTTPS는 HTTP에 TLS 암호화를 적용한 프로토콜로, 도청과 중간자 공격(MITM)을 방지합니다.

TLS Handshake는 TCP 3-way handshake 이후 진행되며, 크게 세 가지 목적을 가집니다. 첫째, 서버가 CA가 서명한 인증서를 전달해 신뢰할 수 있는 서버임을 인증합니다. 둘째, 사용할 암호화 알고리즘(Cipher Suite)을 협상합니다. 셋째, 이후 통신에 쓸 대칭 세션 키를 안전하게 교환합니다.

이 과정에서 초기 키 교환은 비대칭 암호화를 사용하고, 실제 데이터 통신은 성능이 빠른 대칭 암호화로 전환됩니다.

---

### ⏱️ 1분 버전

HTTPS는 HTTP에 TLS 레이어를 얹은 프로토콜입니다. 기존 HTTP는 평문 전송이라 도청·중간자 공격에 취약한데, TLS 암호화로 이를 해결합니다.

TLS Handshake는 TCP 3-way handshake 완료 후 시작되며 주요 단계는 다음과 같습니다.

1. **ClientHello** : 클라이언트가 지원하는 TLS 버전과 Cipher Suite 목록을 서버에 전달합니다.
2. **ServerHello + Certificate** : 서버는 Cipher Suite를 선택하고, CA 서명이 된 공개키 인증서를 보냅니다.
3. **인증서 검증** : 클라이언트는 브라우저에 내장된 CA 공개키로 인증서의 유효성을 확인합니다.
4. **세션 키 생성** : 검증이 끝나면 Pre-Master Secret을 서버 공개키로 암호화해 전송하거나, Diffie-Hellman 방식으로 세션 키를 협상합니다.

이후 양측이 동일한 대칭 세션 키를 보유하게 되고, 실제 HTTP 데이터는 이 세션 키로 암복호화됩니다.

---

### ⏱️ 1분 10초 버전

HTTPS는 HTTP 위에 TLS(Transport Layer Security) 프로토콜을 추가한 것으로, 세 가지 보안 속성을 제공합니다. **기밀성**(암호화로 도청 방지), **무결성**(데이터 변조 감지), **인증**(서버 신원 확인)입니다.

TLS Handshake는 TCP 3-way handshake 이후에 이루어지며, 현재 표준은 **TLS 1.3**입니다. 세부 흐름은 다음과 같습니다.

| 단계 | 방향 | 내용 |
|------|------|------|
| ClientHello | C→S | TLS 버전, 지원 Cipher Suite, 난수(Client Random) 전송 |
| ServerHello | S→C | Cipher Suite 선택, 난수(Server Random) 전송 |
| Certificate | S→C | CA 서명된 서버 공개키 인증서 전송 |
| 인증서 검증 | Client | 브라우저 내장 CA 루트 인증서로 체인 검증 |
| Key Exchange | C↔S | Pre-Master Secret 전달 또는 ECDHE로 키 협상 |
| Finished | C↔S | 양측이 동일한 Session Key 보유 확인 후 핸드셰이크 종료 |

핵심은 **초기 키 교환 단계에서만 비대칭 암호화(RSA/ECDHE)를 사용**하고, 실제 데이터 통신은 속도가 빠른 **대칭 암호화(AES 등)로 전환**한다는 점입니다. TLS 1.3에서는 핸드셰이크가 1-RTT로 줄고, 재접속 시 0-RTT도 지원해 성능이 크게 개선되었습니다.

</details>
