<details>
<summary>1. 트랜잭션에 대해서 설명해주세요.</summary>

> 트랜잭션은 데이터베이스에서  
하나의 작업 단위로 묶여서 함께 처리되는 연산입니다.

이 개념은  
중간에 일부만 반영되어 데이터 정합성이 깨지는 문제를 막기 위해 필요합니다.

핵심 특징은  
첫째, 전부 성공하거나 전부 실패해야 한다는 점이고,  
둘째, 처리 중 오류가 나면 이전 상태로 되돌릴 수 있다는 점이고,  
셋째, 여러 작업이 동시에 일어나도 일관성을 유지하도록 관리한다는 점입니다.

그래서 실제로는  
계좌 이체, 주문 생성, 결제 처리처럼  
여러 변경이 하나의 흐름으로 정확하게 묶여야 하는 상황에서 사용됩니다.
</details>

<summary>5. 트랜잭션 격리 수준에 대해 설명해주세요.</summary>

면접에서 45초 정도로 짧게 말하면  
“트랜잭션 격리 수준은 여러 트랜잭션이 동시에 실행될 때  
한 트랜잭션이 다른 트랜잭션의 변경 내용을 어디까지 볼 수 있는지 정하는 기준입니다.  
아무 제어 없이 처리하면 더티 리드, 반복 불가능한 읽기, 팬텀 리드 같은 정합성 문제가 발생하기 때문에 필요합니다.  
격리 수준은 Read Uncommitted, Read Committed, Repeatable Read, Serializable 네 단계가 있고,  
아래로 갈수록 동시성은 높지만 정합성은 약하고, 위로 갈수록 정합성은 강하지만 성능 저하가 큽니다.  
즉, 격리 수준은 동시성과 정합성 사이에서 균형을 잡는 장치입니다.”  
정도로 답할 수 있습니다.

1분 정도로 말하면  
“트랜잭션 격리 수준은 동시에 실행되는 트랜잭션들이 서로의 변경 내용을 어느 정도까지 볼 수 있게 할지 정하는 기준입니다.  
이 기준이 없으면 더티 리드, 반복 불가능한 읽기, 팬텀 리드 같은 문제가 발생할 수 있습니다.  
Read Uncommitted는 미커밋 데이터도 읽을 수 있어 문제가 가장 많고,  
Read Committed는 커밋된 데이터만 읽어 더티 리드는 막지만 반복 불가능한 읽기와 팬텀 리드는 남습니다.  
Repeatable Read는 같은 트랜잭션 안에서 같은 데이터를 반복 조회해도 값이 바뀌지 않도록 보장하고,  
Serializable은 트랜잭션을 순차 실행처럼 처리해서 정합성을 가장 강하게 보장합니다.  
결국 격리 수준은 정합성을 얼마나 우선할지, 아니면 성능과 동시성을 더 우선할지를 결정하는 기준입니다.”  
이렇게 말하면 됩니다.

1분 30초 정도로 자세히 말하면  
“트랜잭션 격리 수준은 여러 트랜잭션이 동시에 실행될 때  
한 트랜잭션이 다른 트랜잭션의 변경 내용을 어디까지 볼 수 있는지를 정하는 기준입니다.  
동시에 여러 사용자가 같은 데이터를 읽고 수정하는 환경에서는  
이 기준이 없으면 정합성 문제가 쉽게 발생합니다.  
대표적으로 더티 리드는 아직 커밋되지 않은 데이터를 읽는 문제이고,  
반복 불가능한 읽기는 같은 트랜잭션 안에서 같은 쿼리를 두 번 실행했을 때 값이 달라지는 문제이고,  
팬텀 리드는 같은 조건으로 조회했을 때 행의 개수 자체가 달라지는 문제입니다.

격리 수준은 낮은 단계부터 네 가지로 나뉩니다.  
먼저 Read Uncommitted는 미커밋 데이터까지 읽을 수 있어서  
세 가지 문제가 모두 발생할 수 있습니다.  
다음 Read Committed는 커밋된 데이터만 읽게 해서 더티 리드는 막지만,  
중간에 다른 트랜잭션이 값을 수정하거나 새 행을 추가하면  
반복 불가능한 읽기와 팬텀 리드는 발생할 수 있습니다.  
그다음 Repeatable Read는 같은 트랜잭션 안에서 이미 읽은 데이터의 값이 변하지 않도록 보장해서  
반복 불가능한 읽기를 방지합니다.  
이론적으로는 팬텀 리드 가능성이 있지만,  
DBMS 구현에 따라 상당 부분 방어되기도 합니다.  
마지막 Serializable은 트랜잭션을 마치 순서대로 하나씩 실행한 것처럼 처리해서  
세 가지 문제를 모두 막을 수 있지만,  
그만큼 락 경합과 성능 저하가 가장 큽니다.

실무에서는 항상 가장 높은 격리 수준을 쓰는 것이 정답은 아닙니다.  
예를 들어 계좌 이체, 재고 차감, 결제처럼 데이터 정합성이 매우 중요한 기능은  
더 강한 격리 수준이 필요할 수 있습니다.  
반면 게시글 조회나 통계 조회처럼 동시성과 성능이 더 중요한 경우에는  
Read Committed 같은 수준으로 타협하는 경우가 많습니다.  
즉, 격리 수준은 높을수록 안전하지만 느려지고,  
낮을수록 빠르지만 정합성 문제가 생길 수 있기 때문에  
서비스 특성에 맞게 선택하는 것이 중요합니다.”  
이렇게 답하면 됩니다.

<summary>3. 정규화에 대해서 설명해주세요.</summary>

> 정규화는 데이터베이스에서 중복을 줄이고 이상 현상을 방지하도록 테이블을 구조화하는 과정입니다.

이 개념은 데이터가 중복되면 수정, 삽입, 삭제 과정에서 불일치가 생길 수 있기 때문에 필요합니다.

핵심 특징은 크게 세 가지입니다.

첫째, 데이터 중복을 최소화합니다.  
같은 정보를 여러 곳에 반복해서 저장하지 않도록 테이블을 나눕니다.

둘째, 데이터 일관성을 높입니다.  
한 곳만 수정해도 전체 데이터에 동일하게 반영되도록 구조를 정리할 수 있습니다.

셋째, 이상 현상을 줄입니다.  
즉 삽입 이상, 수정 이상, 삭제 이상 같은 문제를 예방할 수 있습니다.

예를 들어 회원 이름과 주문 정보를 하나의 테이블에 다 넣으면 회원 정보가 반복 저장될 수 있습니다.  
이 경우 회원 테이블과 주문 테이블을 분리하면 중복을 줄이고 관리가 쉬워집니다.

그래서 정규화는 데이터를 더 안정적이고 일관되게 관리하기 위한 설계 방식이라고 볼 수 있습니다.  
다만 정규화를 너무 많이 하면 조회 시 조인이 많아질 수 있어서, 실제 서비스에서는 성능을 고려해 반정규화도 함께 검토합니다.
</details>

<summary>4. JOIN에 대해서 설명해주세요.</summary>

> JOIN은 관계형 데이터베이스에서  
여러 테이블을 공통된 컬럼 기준으로 연결해서 조회하는 기능입니다.

이 개념은  
정규화로 인해 나뉘어 저장된 데이터를  
하나의 결과처럼 조회하기 위해 필요합니다.

핵심 특징은  
첫째, INNER JOIN은 양쪽에 모두 있는 데이터만 조회하고,  
둘째, LEFT JOIN은 왼쪽 테이블 기준으로 모두 조회하고,  
셋째, 상황에 따라 RIGHT JOIN, FULL JOIN도 사용할 수 있다는 점입니다.

그래서 실제로는  
회원과 주문, 게시글과 댓글처럼  
서로 연관된 데이터를 함께 조회할 때 사용됩니다.

</details>

<summary>5. RDBMS와 NoSQL의 차이에 대해 설명해주세요.</summary>
면접에서 45초 정도로 짧게 말하면  
“RDBMS는 테이블 기반으로 데이터를 저장하고,  
FK와 JOIN을 통해 관계를 정의하며 ACID 트랜잭션으로 강한 일관성을 보장합니다.  
대표적으로 MySQL, PostgreSQL이 있습니다.  
반면 NoSQL은 key-value나 document 같은 다양한 형태로 데이터를 저장하고,  
스키마가 유연하며 수평 확장에 유리합니다.  
대신 보통 RDBMS보다 일관성보다는 가용성과 성능을 더 우선하는 경우가 많습니다.  
그래서 복잡한 관계와 트랜잭션이 중요하면 RDBMS를,  
대용량 트래픽과 유연한 구조가 중요하면 NoSQL을 선택합니다.”  
정도로 답할 수 있습니다.

1분 정도로 말하면  
“RDBMS는 테이블 기반 구조라 데이터 관계를 FK와 JOIN으로 명확하게 표현할 수 있고,  
SQL을 통해 복잡한 조회도 잘 처리할 수 있습니다.  
또한 ACID 트랜잭션을 지원해서 금융, 결제, 주문처럼  
정합성이 중요한 데이터에 적합합니다.  
반면 NoSQL은 key-value, document, column 등 다양한 방식으로 데이터를 저장하며,  
스키마가 유연하고 수평 확장이 쉬워서 대규모 트래픽 환경에 유리합니다.  
대신 일반적으로는 RDBMS보다 강한 일관성보다는 가용성과 성능을 더 우선하는 경우가 많습니다.  
실무에서는 핵심 도메인 데이터는 RDBMS로 관리하고,  
세션이나 캐시, 로그처럼 빠른 처리와 확장성이 중요한 데이터는 NoSQL로 함께 사용하는 경우가 많습니다.”  
이렇게 말하면 됩니다.

1분 30초 정도로 자세히 말하면  
“RDBMS와 NoSQL의 가장 큰 차이는 데이터 모델링 방식과 일관성, 확장 전략에 있습니다.  
먼저 RDBMS는 테이블 기반 구조를 사용하고,  
각 테이블 간 관계를 FK로 정의하며 JOIN을 통해 연관 데이터를 조합해서 조회합니다.  
그래서 데이터 간 관계가 복잡하고 정합성이 중요한 시스템에 적합합니다.  
또한 SQL이라는 표준 질의 언어를 사용해 다양한 조건으로 유연하게 조회할 수 있고,  
무엇보다 ACID 트랜잭션을 통해 원자성, 일관성, 격리성, 지속성을 강하게 보장합니다.  
이 때문에 금융, 주문, 결제처럼 데이터가 틀리면 큰 문제가 되는 도메인에서 많이 사용됩니다.  
대표적인 예로는 MySQL과 PostgreSQL이 있습니다.  
다만 전통적으로는 스케일 아웃보다 스케일 업 중심이라  
트래픽이 매우 커지면 확장에 한계가 생길 수 있습니다.

반면 NoSQL은 관계형 테이블 구조 대신  
key-value, document, column, graph 같은 다양한 모델을 사용합니다.  
고정 스키마가 없거나 훨씬 느슨해서  
서비스 초기처럼 요구사항이 자주 바뀌거나 데이터 구조가 자주 변하는 환경에 잘 맞습니다.  
또한 분산 저장과 수평 확장에 유리해서  
대용량 트래픽 처리나 빠른 응답이 중요한 시스템에서 강점을 가집니다.  
대표적으로 Redis, MongoDB, Cassandra 등이 있습니다.  
대신 많은 NoSQL 시스템은 ACID보다는 BASE에 가까운 철학을 가지며,  
즉시 일관성보다는 가용성과 최종 일관성을 더 우선하는 경우가 많습니다.

실무에서는 둘 중 하나만 고집하기보다  
각각의 강점에 맞게 함께 사용하는 경우가 많습니다.  
예를 들어 회원, 주문, 결제 같은 핵심 도메인 데이터는 RDBMS로 관리하고,  
세션, 캐시, 로그, 실시간 랭킹처럼 빠른 처리와 확장성이 중요한 데이터는  
Redis나 MongoDB 같은 NoSQL로 분리해서 운영합니다.  
즉, 관계와 정합성이 핵심이면 RDBMS,  
유연한 구조와 대규모 확장성이 핵심이면 NoSQL이 더 적합하다고 정리할 수 있습니다.”  
이렇게 답하면 됩니다.

<summary>
## Q. 데이터베이스에서 인덱스를 사용하는 이유 및 장단점에 대해 설명해주세요.

</summary>

---

### ⏱️ 45초 버전

인덱스는 테이블의 특정 컬럼에 대해 **B-Tree 자료구조** 기반으로 별도 생성되는 탐색 구조로, 
Full Table Scan 없이 원하는 행을 빠르게 찾을 수 있게 해줍니다.
**장점**으로는 SELECT 성능 향상, B-Tree 정렬 특성으로 인한 ORDER BY 최적화가 있습니다. PK, FK, Unique 제약 조건에는 DDL 시 자동으로 생성됩니다.
**단점**으로는 인덱스 저장을 위한 추가 메모리 사용과, INSERT·UPDATE·DELETE 시 인덱스도 함께 갱신되어 **쓰기 성능이 저하**된다는 점입니다. 
따라서 **카디널리티(중복도)가 낮은 컬럼**, 즉 값의 종류가 다양한 컬럼에 적용할 때 효과적입니다.

---

### ⏱️ 1분 버전

인덱스는 특정 컬럼의 값과 행 위치를 **B-Tree 구조**로 미리 정렬·저장해, 조회 시 Full Table Scan 대신 **O(log N)** 탐색이 가능하게 하는 구조입니다.

**장점**은 다음과 같습니다.
1. **SELECT 성능 향상** : 대용량 테이블에서 특정 행을 빠르게 탐색
2. **ORDER BY 최적화** : B-Tree의 정렬 특성으로 별도 정렬 연산 생략 가능
3. **자동 생성** : PK, Unique, FK 제약 조건 선언 시 인덱스 자동 생성

**단점**은 다음과 같습니다.
1. **추가 저장 공간** : 인덱스 자체가 별도 메모리·디스크 공간 사용
2. **쓰기 성능 저하** : INSERT·UPDATE·DELETE 시 인덱스도 함께 갱신 필요

따라서 **카디널리티가 높은 컬럼**(값의 중복도가 낮은 컬럼)에 인덱스를 적용해야 효과적이며, 과도한 인덱스 생성은 오히려 전체 성능을 저하시킬 수 있습니다.

---

### ⏱️ 1분 10초 버전

인덱스는 테이블 컬럼에 대해 **B-Tree 구조**로 정렬·저장된 별도 탐색 객체로, 조회 시 Full Table Scan을 피하고 **O(log N)** 탐색을 가능하게 합니다.

| 구분 | 내용 |
|------|------|
| **장점** | SELECT 속도 향상, ORDER BY 최적화, 집계 연산 효율화 |
| **단점** | 추가 저장 공간 소비, DML(INSERT·UPDATE·DELETE) 시 인덱스 갱신 비용 발생 |

인덱스 종류도 중요한데, **클러스터형 인덱스**는 데이터 자체가 정렬되어 저장(PK가 대표적)되고, **비클러스터형 인덱스**는 별도 인덱스 페이지에 포인터로 저장됩니다.

인덱스를 **잘 설계해야 하는 기준**은 다음과 같습니다.

- **카디널리티**가 높을수록(중복값이 적을수록) 인덱스 효과가 큼
- **복합 인덱스** 사용 시 선두 컬럼 기준으로 쿼리가 작성되어야 인덱스가 활용됨
- 조회 비율이 압도적으로 높은 컬럼에 우선 적용하고, DML이 빈번한 컬럼은 인덱스 비용을 신중히 고려해야 함

옵티마이저가 인덱스보다 Full Scan이 유리하다고 판단하면 인덱스를 타지 않을 수도 있으므로, 실행 계획(`EXPLAIN`)을 통한 검증도 중요합니다.
