<summary>1. Spring DI/IoC는 어떻게 동작하나요?</summary>

면접에서 45초 정도로 짧게 말하면  
“IoC는 객체 생성과 의존성 관리를 개발자가 아니라 Spring 컨테이너가 담당하는 개념입니다.  
DI는 이를 구현하는 방식으로, 필요한 객체를 외부에서 주입받는 것입니다.  
Spring은 실행 시 컴포넌트 스캔으로 Bean을 등록하고,  
생성자 주입이나 `@Autowired`를 통해 필요한 곳에 의존성을 연결합니다.  
이 덕분에 객체 간 결합도가 낮아지고 테스트나 구현체 교체가 쉬워집니다.”  
정도로 답할 수 있습니다.

1분 정도로 말하면  
“IoC는 제어의 역전으로, 객체 생성과 생명주기 관리 권한이 개발자 코드가 아니라 Spring 컨테이너에 있는 구조입니다.  
DI는 IoC를 구현하는 방식으로, 객체가 필요한 의존 객체를 직접 만들지 않고 외부에서 주입받도록 합니다.  
Spring은 실행 시 `@Component`, `@Service`, `@Repository` 같은 어노테이션이 붙은 클래스를 스캔해서 Bean으로 등록하고 컨테이너가 관리합니다.  
이후 생성자 주입, 세터 주입, 필드 주입 같은 방식으로 의존성을 연결하는데,  
실무에서는 불변성과 테스트 용이성 때문에 생성자 주입을 가장 권장합니다.  
이 구조 덕분에 구현체보다 인터페이스에 의존하기 쉬워져서 결합도가 낮아지고, 테스트 시 Mock 객체로 교체하기도 쉬워집니다.”  
이렇게 말하면 됩니다.

1분 30초 정도로 자세히 말하면  
“IoC는 제어의 역전이라는 개념으로, 객체를 생성하고 의존 관계를 맺고 생명주기를 관리하는 권한이 개발자가 아니라 Spring 컨테이너에 있는 구조를 말합니다.  
전통적인 방식에서는 개발자가 필요한 객체를 직접 `new`로 생성하고 연결해야 했지만,  
Spring에서는 이 역할을 컨테이너가 대신 수행합니다.  
DI는 그런 IoC를 구현하는 핵심 방식으로, 객체가 자신이 사용할 의존 객체를 직접 생성하지 않고  
외부에서 주입받게 만드는 방식입니다.

동작 과정을 보면 먼저 애플리케이션이 시작될 때  
Spring이 컴포넌트 스캔이나 `@Configuration`과 `@Bean` 설정을 통해  
관리 대상 객체를 Bean으로 등록합니다.  
예를 들어 `@Component`, `@Service`, `@Repository`, `@Controller`가 붙은 클래스들이  
ApplicationContext에 등록되고, 컨테이너가 이를 싱글톤 중심으로 관리합니다.  
그 다음 Spring은 각 Bean의 생성자나 필드, 세터를 보고  
어떤 의존 객체가 필요한지 분석한 뒤  
적절한 Bean을 찾아 자동으로 연결해줍니다.

주입 방식은 크게 생성자 주입, 세터 주입, 필드 주입이 있습니다.  
이 중 생성자 주입은 객체 생성 시점에 의존성이 확정되기 때문에  
불변성을 유지하기 쉽고, 필수 의존성을 강제할 수 있어서 가장 권장됩니다.  
세터 주입은 선택적 의존성에 사용할 수 있고,  
필드 주입은 코드가 간단해 보이지만 테스트가 어렵고 객체 구조가 명확히 드러나지 않아서 실무에서는 잘 쓰지 않습니다.

이 구조의 가장 큰 장점은  
객체가 구현체를 직접 만들지 않으므로 결합도가 낮아진다는 점입니다.  
즉, 인터페이스에 의존하도록 설계하기 쉬워지고,  
새 구현체로 교체하거나 테스트에서 Mock 객체를 주입하기도 쉬워집니다.  
그래서 결과적으로 OCP와 DIP 같은 객체지향 원칙을 자연스럽게 지키기 쉬운 구조가 됩니다.”  
이렇게 답하면 됩니다.

<summary>2. JPA 영속성 컨텍스트의 이점 5가지를 설명해주세요.</summary>

면접에서 45초 정도로 짧게 말하면  
“영속성 컨텍스트는 JPA가 엔티티를 관리하는 논리적 공간이고,  
대표적인 이점은 다섯 가지입니다.  
첫째 1차 캐시로 같은 트랜잭션 안에서 재조회 시 DB를 다시 가지 않습니다.  
둘째 동일성 보장으로 같은 ID의 엔티티는 같은 인스턴스를 반환합니다.  
셋째 쓰기 지연으로 SQL을 모았다가 커밋 시점에 실행합니다.  
넷째 변경 감지로 엔티티 수정 시 update SQL을 자동 생성합니다.  
다섯째 지연 로딩으로 연관 엔티티를 실제 사용하는 시점에 조회해서 성능을 최적화합니다.”  
정도로 답할 수 있습니다.

1분 정도로 말하면  
“영속성 컨텍스트는 JPA가 엔티티를 저장하고 관리하는 공간으로,  
객체와 DB 사이를 효율적으로 중개하는 역할을 합니다.  
핵심 이점은 다섯 가지인데,  
첫째 1차 캐시로 같은 트랜잭션 안에서 동일 엔티티를 재조회할 때 DB 접근을 줄일 수 있고,  
둘째 동일성 보장으로 같은 ID의 엔티티는 항상 같은 인스턴스를 반환합니다.  
셋째 쓰기 지연으로 persist 시점에 즉시 SQL을 보내지 않고 커밋 시점에 모아서 실행합니다.  
넷째 변경 감지로 엔티티 필드만 수정해도 JPA가 변경 사항을 감지해 update SQL을 자동 생성합니다.  
다섯째 지연 로딩으로 연관 엔티티를 실제 사용하는 시점에 조회해서 불필요한 쿼리를 줄일 수 있습니다.”  
이렇게 말하면 됩니다.

1분 30초 정도로 자세히 말하면  
“영속성 컨텍스트는 JPA가 엔티티를 저장하고 관리하는 논리적 공간으로,  
단순히 SQL을 실행하는 기능을 넘어서 객체와 데이터베이스 사이를 효율적으로 연결해주는 역할을 합니다.  
핵심 이점은 다섯 가지입니다.

첫째, 1차 캐시입니다.  
같은 영속성 컨텍스트 안에서 동일한 엔티티를 다시 조회하면  
DB에 재조회하지 않고 메모리에 있는 엔티티를 반환합니다.  
그래서 같은 트랜잭션 내 조회 성능을 높일 수 있습니다.

둘째, 동일성 보장입니다.  
같은 ID로 조회했을 때 단순히 값이 같은 객체를 주는 것이 아니라  
실제로 같은 인스턴스를 반환합니다.  
즉, 같은 트랜잭션 안에서는 엔티티의 동일성을 보장할 수 있습니다.

셋째, 쓰기 지연입니다.  
`persist()`를 호출해도 INSERT SQL이 즉시 DB로 전송되지 않고,  
쓰기 지연 SQL 저장소에 모였다가 트랜잭션 커밋 시점에 한 번에 실행됩니다.  
이 방식은 네트워크 비용을 줄이고 배치 처리에도 유리합니다.

넷째, 변경 감지입니다.  
영속 상태의 엔티티를 조회한 뒤 값을 변경하면  
JPA가 최초 상태를 스냅샷으로 저장해두었다가  
트랜잭션이 끝날 때 현재 상태와 비교합니다.  
그 결과 변경된 필드가 있으면 자동으로 UPDATE SQL을 생성해서 실행합니다.  
그래서 개발자는 setter로 값만 바꿔도 되고, 별도의 update 메서드를 직접 호출하지 않아도 됩니다.

다섯째, 지연 로딩입니다.  
연관 엔티티를 조회할 때 처음부터 전부 가져오는 것이 아니라  
프록시 객체로 보관해두고 실제 접근 시점에 쿼리를 실행합니다.  
이 덕분에 불필요한 조인과 조회를 줄일 수 있어 성능 최적화에 도움이 됩니다.  
다만 잘못 사용하면 N+1 문제가 생길 수 있어서 fetch join 같은 전략과 함께 고려해야 합니다.

결국 영속성 컨텍스트는  
객체를 단순 저장하는 수준이 아니라  
조회, 동일성, 변경 추적, 쓰기 시점 제어, 지연 조회까지 관리해주기 때문에  
JPA를 객체지향적으로 사용할 수 있게 만드는 핵심 개념이라고 볼 수 있습니다.”  
이렇게 답하면 됩니다.

<summary>3. 인터페이스와 추상클래스의 차이점에 대해 설명해주세요.</summary>

면접에서 45초 정도로 짧게 말하면  
“인터페이스와 추상클래스는 둘 다 공통 동작을 추상화하지만 목적이 다릅니다.  
인터페이스는 ‘무엇을 할 수 있는가’라는 기능 규약을 정의하는 데 적합하고,  
추상클래스는 공통 상태와 기본 동작을 함께 재사용하는 데 적합합니다.  
또 자바에서 인터페이스는 다중 구현이 가능하지만,  
추상클래스는 단일 상속만 가능합니다.  
그래서 역할 조합이 중요하면 인터페이스를,  
공통 상태와 구현 재사용이 중요하면 추상클래스를 선택합니다.”  
정도로 답할 수 있습니다.

1분 정도로 말하면  
“인터페이스와 추상클래스는 모두 공통 동작을 추상화해 유연한 설계를 가능하게 하지만,  
목적과 구조에서 차이가 있습니다.  
인터페이스는 ‘무엇을 할 수 있는가’를 정의하는 기능 규약에 가깝고,  
구현체 간 결합을 낮추고 교체 가능성을 높이는 데 유리합니다.  
반면 추상클래스는 공통 상태와 기본 구현을 함께 재사용하는 데 적합합니다.  
자바는 클래스 단일 상속만 허용하지만 인터페이스는 여러 개를 동시에 구현할 수 있어서  
여러 역할을 조합하기에 좋습니다.  
또 인터페이스는 규약 중심이고,  
추상클래스는 필드, 생성자, 구현 메서드를 가질 수 있다는 차이도 있습니다.”  
이렇게 말하면 됩니다.

1분 30초 정도로 자세히 말하면  
“인터페이스와 추상클래스는 둘 다 공통 동작을 추상화해 유연한 설계를 가능하게 하지만,  
본질적으로는 추상화의 목적이 다릅니다.  
먼저 인터페이스는 ‘무엇을 할 수 있는가’에 초점을 맞춥니다.  
즉, 특정 객체가 제공해야 하는 기능이나 역할의 규약을 정의하는 데 적합합니다.  
구현체 내부가 어떻게 생겼는지와는 별개로,  
같은 메서드 규약만 맞추면 동일한 방식으로 다룰 수 있기 때문에  
다형성과 DIP를 활용하기에 좋습니다.

반면 추상클래스는 ‘공통 상태와 동작을 어떻게 공유할 것인가’에 더 초점을 둡니다.  
즉, 하위 클래스들이 공통으로 가져야 할 필드, 생성자, 기본 메서드 구현을 함께 묶어  
재사용성을 높이는 데 적합합니다.  
그래서 단순히 규약만 정하는 것이 아니라  
일부 공통 로직을 상위 클래스에서 직접 제공하고 싶을 때 많이 사용합니다.

구조적으로 보면  
인터페이스는 자바에서 여러 개를 동시에 구현할 수 있어서  
역할을 조합하는 데 유리합니다.  
예를 들어 어떤 객체가 결제도 가능하고 로깅도 가능하고 알림도 가능하다면  
각 역할을 인터페이스로 분리해서 조합할 수 있습니다.  
반면 추상클래스는 클래스 상속이기 때문에 단일 상속만 가능해서  
계층 구조를 잘못 설계하면 유연성이 떨어질 수 있습니다.

또 구현 범위도 다릅니다.  
인터페이스는 기본적으로 추상 메서드 중심이지만,  
Java 8 이후에는 default 메서드를 통해 기본 구현도 일부 제공할 수 있습니다.  
그래도 여전히 핵심 목적은 규약 정의입니다.  
반면 추상클래스는 일반 필드, 생성자, 구현된 메서드, 추상 메서드를 모두 가질 수 있어서  
상태를 공유하는 계층 구조를 설계할 때 더 적합합니다.

예를 들어 실무에서 결제 수단처럼  
카드 결제, 계좌 이체, 간편 결제 등 여러 구현체를 같은 방식으로 다루고 싶다면  
`Payable` 같은 인터페이스로 `pay()` 규약을 정의하는 것이 적절합니다.  
반면 이름, 나이 같은 공통 필드와 기본 행동을 여러 객체가 함께 가져야 한다면  
`Animal` 같은 추상클래스를 두는 것이 더 자연스럽습니다.  
Spring에서도 인터페이스 기반 DI를 많이 사용하는 이유가  
바로 구현체 교체와 테스트 대체가 쉬워지기 때문입니다.”  
이렇게 답하면 됩니다.

<details> <summary>4. JWT에 대해서 간단히 설명해주세요.</summary>

JWT는 사용자 정보를 JSON 형태로 담고, 이에 서명을 붙여 만든 토큰 기반 인증 방식입니다.
이 개념은 서버가 로그인 상태를 직접 저장하지 않고도 사용자를 식별하고 인증하기 위해 필요합니다.
핵심 특징은 크게 세 가지입니다.

첫째, Header, Payload, Signature의 세 부분으로 구성됩니다.
Header에는 토큰 타입과 알고리즘 정보가 들어가고, Payload에는 사용자 정보나 권한 같은 데이터가 들어가며, Signature는 위변조 여부를 검증하는 역할을 합니다.

둘째, 서버가 상태를 저장하지 않는 무상태 인증 방식에 적합합니다.
즉 세션처럼 로그인 정보를 서버 메모리나 저장소에 따로 유지하지 않아도 됩니다.

셋째, 서명을 통해 토큰이 중간에 변조되었는지 확인할 수 있습니다.
다만 Payload 자체는 암호화가 아니라 인코딩이기 때문에 민감한 정보는 넣지 않는 것이 중요합니다.

예를 들어 사용자가 로그인하면 서버가 JWT를 발급하고, 이후 요청마다 클라이언트가 이 토큰을 함께 보내면 서버는 토큰을 검증해서 인증을 처리할 수 있습니다.

그래서 JWT는 확장성이 좋고 분산 환경에 잘 어울리는 인증 방식입니다.
다만 토큰이 탈취되면 위험하므로 만료 시간, 재발급 방식, 저장 위치를 함께 신중하게 설계해야 합니다.

</details>


<details> <summary>5. 정적(static)이란 무엇인가요?</summary>

static은 객체가 아니라 클래스 자체에 속하는 요소를 의미합니다.
이 개념은 여러 객체가 공통으로 사용해야 하는 값이나 기능을 하나로 관리하기 위해 필요합니다.
핵심 특징은 크게 세 가지입니다.

첫째, 객체를 생성하지 않아도 접근할 수 있습니다.
즉 클래스명만으로 바로 사용할 수 있습니다.

둘째, 모든 인스턴스가 같은 값을 공유합니다.
그래서 static 변수는 객체마다 따로 생성되지 않고 클래스 기준으로 하나만 존재합니다.

셋째, 공통 기능이나 상수를 표현할 때 자주 사용됩니다.
예를 들어 유틸 메서드나 애플리케이션 전체에서 동일하게 써야 하는 상수에 적합합니다.

예를 들어 Math.max()는 객체를 만들지 않고 바로 호출할 수 있는 static 메서드입니다.
또 public static final 상수는 프로그램 전체에서 공통 기준값으로 자주 사용됩니다.

그래서 static은 공통 자원이나 공통 기능을 클래스 단위로 관리하기 위한 키워드라고 볼 수 있습니다.
다만 상태를 static으로 많이 두면 결합도가 높아지고 테스트가 어려워질 수 있어서 신중하게 사용해야 합니다.

</details>
<details>
<summary>6. 비대칭키 암호화, 대칭키 암호화에 대해 간단히 설명해주세요.</summary>

> 대칭키 암호화는  
암호화와 복호화에 같은 키를 사용하는 방식입니다.

이 방식은  
구조가 단순하고 속도가 빨라서  
대용량 데이터를 처리할 때 유리합니다.

반면 비대칭키 암호화는  
공개키와 개인키처럼 서로 다른 키를 사용하는 방식입니다.

이 방식은  
키를 안전하게 주고받는 데 유리하지만,  
대칭키 방식보다 상대적으로 느립니다.

그래서 실제로는  
비대칭키로 키를 안전하게 교환한 뒤,  
실제 데이터 통신은 대칭키로 처리하는 방식이 많이 사용됩니다.

</details>

<details>
<summary>7. 클래스는 무엇이고 객체는 무엇인가요?</summary>

> 클래스는 객체를 만들기 위한  
설계도 또는 틀입니다.

이 개념은  
공통 속성과 동작을 미리 정의해두고  
여러 개의 실제 대상을 만들기 위해 필요합니다.

객체는  
클래스를 바탕으로 실제 생성된 구체적인 인스턴스입니다.

핵심 특징은  
클래스는 정의이고,  
객체는 실제 메모리에 올라가 동작하는 실체라는 점입니다.

그래서 실제로는  
Car라는 클래스가 있으면,  
myCar, yourCar 같은 실제 생성된 대상이 객체라고 볼 수 있습니다.

</details>

<details>
<summary>6. 객체지향에 대해서 설명해주세요.</summary>

면접에서 45초 정도로 짧게 말하면  
“객체지향은 상태와 행동을 객체로 묶고 객체 간 협력으로 동작하는 프로그래밍 패러다임입니다.  
핵심 특성은 캡슐화, 상속, 다형성, 추상화입니다.  
캡슐화는 내부 구현을 숨기고 필요한 인터페이스만 노출하는 것이고,  
상속은 부모의 속성과 동작을 재사용하는 것이며,  
다형성은 같은 인터페이스로 여러 구현체를 다룰 수 있는 것이고,  
추상화는 공통 개념을 인터페이스나 추상 클래스로 표현하는 것입니다.  
이 덕분에 변경에 유연하고 확장하기 쉬운 코드를 만들 수 있습니다.”  
정도로 답할 수 있습니다.

1분 정도로 말하면  
“객체지향은 절차적으로 기능을 나열하는 방식과 달리,  
관련된 상태와 행동을 객체로 묶고 객체 간 협력으로 문제를 해결하는 방식입니다.  
핵심 특성은 네 가지인데,  
캡슐화는 내부 구현을 숨기고 인터페이스만 노출해서 외부 의존을 줄이는 것이고,  
상속은 공통 속성과 동작을 재사용하는 것이고,  
다형성은 같은 타입으로 여러 구현체를 교체할 수 있게 하는 것이고,  
추상화는 복잡한 구현에서 핵심 역할만 뽑아내는 것입니다.  
실무에서는 특히 다형성과 추상화가 중요하고,  
이 덕분에 기존 코드를 크게 바꾸지 않고 새로운 구현체를 추가하는 식으로 확장할 수 있습니다.  
Spring의 DI도 이런 객체지향 원칙 위에서 많이 활용됩니다.”  
이렇게 말하면 됩니다.

1분 30초 정도로 자세히 말하면  
“객체지향 프로그래밍은 관련된 데이터와 기능을 객체라는 단위로 묶고,  
객체들이 서로 메시지를 주고받으며 협력해서 문제를 해결하는 프로그래밍 패러다임입니다.  
절차지향이 동작 순서를 함수 중심으로 나열하는 방식이라면,  
객체지향은 역할과 책임을 객체에 나눠서 설계한다는 점이 다릅니다.

핵심 특성은 네 가지입니다.  
먼저 캡슐화는 객체 내부 상태와 구현 세부사항을 외부에 숨기고  
필요한 기능만 메서드로 노출하는 것입니다.  
이 덕분에 내부 구현이 바뀌어도 외부 코드에 미치는 영향을 줄일 수 있습니다.  
다음으로 상속은 부모 클래스의 속성과 동작을 자식 클래스가 물려받아 재사용하는 개념입니다.  
다만 상속은 편리하지만 부모와 자식이 강하게 결합될 수 있어서  
실무에서는 무조건 상속보다는 조합과 인터페이스 기반 설계를 더 신중하게 선택합니다.

세 번째는 다형성입니다.  
다형성은 같은 인터페이스를 통해 서로 다른 구현체를 동일한 방식으로 사용할 수 있는 특성입니다.  
예를 들어 결제 기능을 `PaymentService` 인터페이스로 추상화해두면  
카드 결제, 간편 결제, 테스트용 Mock 구현체를 호출부 수정 없이 교체할 수 있습니다.  
마지막으로 추상화는 복잡한 구현 세부사항에서 핵심 역할만 뽑아내는 것입니다.  
보통 인터페이스나 추상 클래스로 공통 개념을 표현하면서  
구체 구현과 역할을 분리할 때 사용합니다.

이 네 가지 특성 중 실무에서 특히 중요한 것은 다형성과 추상화입니다.  
이 둘을 잘 활용하면 기존 코드를 수정하지 않고 새 구현체를 추가하는 방식으로 확장할 수 있어서  
OCP를 지키기 쉬워지고, 인터페이스에 의존하도록 설계하면 DIP도 실현하기 쉬워집니다.  
Spring에서 인터페이스를 두고 구현체를 DI로 주입받는 구조가 대표적인 예입니다.  
이런 구조 덕분에 테스트에서는 Mock으로 쉽게 교체할 수 있고,  
운영 코드에서도 구현 변경에 유연하게 대응할 수 있습니다.”  
이렇게 답하면 됩니다.

</details>
