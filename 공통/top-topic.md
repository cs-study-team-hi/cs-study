<summary>1. Spring DI/IoC는 어떻게 동작하나요?</summary>
> 트랜잭션은 데이터베이스에서  
하나의 작업 단위로 묶여서 함께 처리되는 연산입니다.

이 개념은  
중간에 일부만 반영되어 데이터 정합성이 깨지는 문제를 막기 위해 필요합니다.

핵심 특징은  
첫째, 전부 성공하거나 전부 실패해야 한다는 점이고,  
둘째, 처리 중 오류가 나면 이전 상태로 되돌릴 수 있다는 점이고,  
셋째, 여러 작업이 동시에 일어나도 일관성을 유지하도록 관리한다는 점입니다.

그래서 실제로는  
계좌 이체, 주문 생성, 결제 처리처럼  
여러 변경이 하나의 흐름으로 정확하게 묶여야 하는 상황에서 사용됩니다.

<summary>2. JPA 영속성 컨텍스트의 이점(5가지)을 설명해주세요.</summary>

영속성 컨텍스트는 엔티티를 저장하고 관리하는 JPA의 핵심 환경입니다.
이 개념이 필요한 이유는 단순히 데이터를 바로 DB에 넣고 꺼내는 수준이 아니라, 애플리케이션 안에서 엔티티의 상태를 일관되게 관리하고, 데이터베이스 접근을 더 효율적으로 처리하기 위해서입니다.
핵심 이점은 대표적으로 다섯 가지가 있습니다.

첫째, 1차 캐시입니다.
영속성 컨텍스트는 같은 트랜잭션 안에서 조회한 엔티티를 메모리에 보관합니다.
그래서 같은 엔티티를 다시 조회하면 DB를 다시 조회하지 않고 1차 캐시에서 가져올 수 있습니다.
즉, 조회 성능을 높이고 같은 요청 안에서 일관된 객체를 사용할 수 있습니다.

둘째, 동일성 보장입니다.
같은 영속성 컨텍스트 안에서 같은 엔티티를 조회하면 항상 같은 인스턴스를 반환합니다.
예를 들어 같은 ID의 Member를 두 번 조회했을 때, 단순히 값이 같은 것이 아니라 실제로 같은 객체 참조를 보장합니다.
이 점은 애플리케이션에서 객체를 다룰 때 매우 중요한 안정성을 제공합니다.

셋째, 트랜잭션을 지원하는 쓰기 지연입니다.
엔티티를 persist 한다고 해서 즉시 insert SQL이 바로 실행되는 것이 아니라, 영속성 컨텍스트가 SQL을 모아두었다가 트랜잭션을 커밋하는 시점에 한 번에 DB로 보냅니다.
이 덕분에 SQL 실행을 모아서 처리할 수 있고, 트랜잭션 단위로 더 효율적인 쓰기 작업이 가능합니다.

넷째, 변경 감지입니다.
영속 상태의 엔티티 값을 변경하면 개발자가 update SQL을 직접 호출하지 않아도, JPA가 변경된 내용을 감지해서 커밋 시점에 update SQL을 생성합니다.
즉, 객체를 수정하는 것만으로도 DB 반영이 가능해져서 객체지향적인 코드 작성이 쉬워집니다.

다섯째, 지연 로딩입니다.
연관된 엔티티를 처음부터 전부 조회하지 않고, 실제로 필요한 시점에 조회할 수 있게 해줍니다.
예를 들어 주문을 조회할 때 회원 정보가 바로 필요하지 않다면 프록시로 들고 있다가, 실제로 회원 정보에 접근하는 시점에 쿼리를 날릴 수 있습니다.
이 방식은 복잡한 연관관계에서 성능 최적화에 도움이 됩니다.
예를 들어 주문과 회원, 상품이 연결된 시스템에서 영속성 컨텍스트가 없다면 조회할 때마다 DB를 반복해서 접근하고, 수정할 때마다 직접 update를 호출해야 합니다.
하지만 영속성 컨텍스트를 사용하면 같은 엔티티는 한 번만 관리하고, 수정은 객체 상태만 바꾸면 되고, 필요한 순간에만 연관 데이터를 가져올 수 있어서 훨씬 객체지향적이고 효율적으로 개발할 수 있습니다.
즉, 영속성 컨텍스트의 5가지 이점은 1차 캐시, 동일성 보장, 쓰기 지연, 변경 감지, 지연 로딩이며, JPA가 단순 SQL 실행 도구가 아니라 객체와 데이터베이스 사이를 효율적으로 관리하게 해주는 핵심 기반이라고 볼 수 있습니다.

<summary>3. 인터페이스와 추상클래스의 차이점에 대해 설명해주세요.</summary>

인터페이스와 추상클래스는 둘 다 공통 동작을 추상화해서 설계를 유연하게 만들기 위한 개념이라는 공통점이 있습니다.
다만 차이는 크게 세 가지가 있습니다.

첫째, 목적 측면에서 인터페이스는 "무엇을 할 수 있는가"에 초점을 맞추고, 추상클래스는 "어떻게 공통 상태와 동작을 공유할 것인가"에 더 가깝습니다.
즉 인터페이스는 기능의 규약을 정의하는 데 적합하고, 추상클래스는 공통된 필드나 기본 구현을 함께 묶어 재사용하는 데 적합합니다.
둘째, 상속 구조 측면에서 자바는 클래스는 단일 상속만 가능하지만 인터페이스는 여러 개를 구현할 수 있습니다.
그래서 인터페이스는 여러 역할을 조합하기 좋고, 추상클래스는 하나의 계층 구조를 잡아가는 데 더 어울립니다.
셋째, 구현과 상태 측면에서 인터페이스는 기본적으로 상수와 추상 메서드 중심으로 규약을 정의하고, 추상클래스는 일반 필드, 생성자, 일반 메서드, 추상 메서드를 함께 가질 수 있습니다.
즉 추상클래스는 상태를 가지면서 일부 공통 로직을 미리 구현해둘 수 있지만, 인터페이스는 상대적으로 구현체 간 결합을 낮추고 역할 중심으로 설계하기 좋습니다.
예를 들어 결제 시스템에서 카드 결제, 카카오페이 결제, 계좌이체 결제가 있다면 Payable 같은 인터페이스를 두고 pay() 규약을 맞추는 방식이 적절합니다.
반면 동물이라는 공통 개념 아래 이름, 나이 같은 공통 필드와 eat() 같은 공통 동작을 함께 묶고 싶다면 Animal 추상클래스를 두는 방식이 더 적절합니다.

그래서 공통 규약과 역할의 조합이 중요하면 인터페이스를 선택하고, 공통 상태와 구현을 함께 재사용해야 하면 추상클래스를 선택합니다.
