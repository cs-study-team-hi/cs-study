<summary>2. 컨텍스트 스위칭에 대해 설명해보세요.</summary>
> 컨텍스트 스위칭은 CPU가 현재 실행 중인 작업의 상태를 저장하고,
다른 작업의 상태를 불러와 실행 대상을 바꾸는 과정입니다.

이 개념은
여러 프로세스나 스레드를 하나의 CPU 또는 제한된 CPU 자원 위에서
번갈아 실행해야 하는 문제를 해결하기 위해 필요합니다.

핵심 특징은
첫째, 현재 작업의 레지스터 값, 프로그램 카운터, 스택 정보 같은 실행 문맥을 저장하고 복원해야 한다는 점이고,
둘째, 이 과정에서 실제 업무를 처리하는 것이 아니라 전환 작업 자체에 비용이 든다는 점이고,
셋째, 멀티태스킹 환경에서 공정한 CPU 분배와 동시성 처리의 기반이 된다는 점입니다.

그래서 실제로는
운영체제가 여러 프로세스를 번갈아 실행할 때,
또는 멀티스레드 환경에서 실행 흐름을 전환할 때 사용됩니다.

2. 동기와 비동기의 차이(블로킹, 넌블로킹) / 장단점에 대해 설명해보세요.
둘 다 작업을 처리하는 흐름을 제어하기 위한 방식이라는 공통점이 있습니다.
다만 차이는 크게 세 가지가 있습니다.

첫째, 동작 방식 측면에서 동기는 요청한 작업이 끝날 때까지 현재 흐름이 결과를 기다리는 방식이고, 비동기는 작업을 요청한 뒤 결과가 끝나기를 기다리지 않고 다음 작업을 이어서 처리하는 방식입니다.
둘째, 블로킹 여부 측면에서 블로킹은 다른 작업의 결과가 나올 때까지 현재 스레드가 멈춰 있는 상태이고, 넌블로킹은 결과가 아직 준비되지 않았더라도 제어권을 바로 돌려받아 다음 작업을 진행할 수 있는 상태입니다. 그래서 보통 동기와 블로킹이 함께 자주 언급되고, 비동기와 넌블로킹도 함께 많이 묶이지만, 엄밀히 말하면 동기/비동기와 블로킹/넌블로킹은 서로 다른 비교 축입니다.
셋째, 성능과 자원 사용 측면에서 동기 + 블로킹 방식은 흐름이 단순해서 이해하고 디버깅하기 쉽지만, 대기 시간이 길어지면 스레드가 놀게 되어 자원 효율이 떨어질 수 있습니다. 반면 비동기 + 넌블로킹 방식은 대기 시간 동안 다른 작업을 처리할 수 있어서 처리량과 자원 활용 면에서 유리하지만, 콜백, 이벤트, Future, Promise 같은 구조가 들어가면서 구현 복잡도와 디버깅 난도가 높아질 수 있습니다.

장단점을 정리하면, 동기 방식은 코드 흐름이 직관적이고 순서를 보장하기 쉬워서 안정적으로 구현하기 좋다는 장점이 있지만, 외부 API 호출이나 파일 입출력처럼 기다리는 시간이 긴 작업이 많아지면 성능이 떨어질 수 있습니다.
비동기 방식은 동시에 더 많은 요청을 효율적으로 처리할 수 있다는 장점이 있지만, 흐름 제어가 복잡하고 예외 처리나 상태 관리가 어려워질 수 있습니다.
그래서 순차적인 처리와 단순한 제어가 중요하면 동기 방식을, 높은 동시성과 자원 효율이 중요하면 비동기 방식을 선택합니다. 
또 요청을 보낸 뒤 스레드를 붙잡아 두지 않는 것이 중요하면 넌블로킹을, 결과가 나올 때까지 확실히 기다리며 처리 순서를 단순하게 가져가고 싶다면 블로킹을 선택합니다.
