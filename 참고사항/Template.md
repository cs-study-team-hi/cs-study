1. CS 면접 공통 답변 템플릿

가장 무난하고 강한 기본형은 이겁니다.
기본 4단 구조
1) 한 줄 정의
   → 이게 무엇인지 바로 말하기
2) 왜 필요한지 / 어떤 문제를 해결하는지
   → 존재 이유 말하기
3) 핵심 동작 원리 / 특징
   → 내부 흐름, 성질, 중요한 포인트
4) 비교 / 장단점 / 사용 예시
   → 비슷한 개념과 차이, 어디에 쓰는지

이걸 문장으로 바꾸면 보통 이렇게 갑니다.

“먼저 정의를 말씀드리면,
A는 ~~입니다.
이건 주로 ~~ 문제를 해결하기 위해 사용됩니다.
핵심 특징은 ~~이고, 동작 방식은 ~~입니다.
비슷한 개념인 B와 비교하면 ~~한 차이가 있고,
그래서 보통 ~~ 상황에서 사용됩니다.”

이 틀은 거의 다 먹힙니다.

# 2)질문 유형별 간단한 정리 ~
## 2-1. “개념 설명형” 템플릿

예: TCP가 뭔가요, 프로세스가 뭔가요, 인덱스가 뭔가요
템플릿
> 정의 → 왜 필요한가 → 핵심 특징 2~3개 → 예시/사용처

말하는 흐름
“A는 ~~입니다.
이 개념은 ~~ 문제를 해결하기 위해 필요합니다.
핵심 특징은 첫째 ~~ , 둘째 ~~ , 셋째 ~~입니다.
그래서 실제로는 ~~ 같은 상황에서 사용됩니다.”

예시

“UDP와 TCP를 설명하세요” 말고 “TCP를 설명하세요”일 때

“TCP는 연결 지향적인 전송 계층 프로토콜입니다.
데이터가 순서대로, 안정적으로 전달되도록 보장하는 것이 목적입니다.
핵심 특징은 연결을 맺고 통신한다는 점, 패킷 순서를 보장한다는 점, 그리고 재전송과 흐름 제어를 지원한다는 점입니다.
그래서 파일 전송이나 웹 통신처럼 신뢰성이 중요한 상황에서 많이 사용됩니다.”

> 2-2. “비교형” 템플릿

예: TCP와 UDP 차이, 프로세스와 스레드 차이, HashMap과 TreeMap 차이

템플릿

공통점 한 줄 → 차이 기준 3개 → 상황별 선택

비교는 그냥 막 말하면 산으로 가기 쉬워서
비교 축을 정해놓고 말해야 합니다.

추천 비교 축:
> 목적
> 동작 방식
> 성능
> 안정성
> 자원 사용
> 사용 상황

말하는 흐름

“둘 다 ~~를 위한 개념이라는 공통점이 있습니다.
다만 차이는 크게 세 가지가 있습니다.
첫째, ~~ 측면에서 A는 ~~이고 B는 ~~입니다.
둘째, ~~ 측면에서 A는 ~~이고 B는 ~~입니다.
셋째, ~~ 측면에서 A는 ~~이고 B는 ~~입니다.
그래서 ~~가 중요하면 A를, ~~가 중요하면 B를 선택합니다.”

TCP vs UDP 예시

“둘 다 전송 계층 프로토콜이라는 공통점이 있습니다.
다만 차이는 크게 세 가지가 있습니다.
첫째, 연결 방식에서 TCP는 연결 지향이고 UDP는 비연결형입니다.
둘째, 신뢰성 측면에서 TCP는 순서 보장과 재전송을 지원하지만 UDP는 그런 보장을 하지 않습니다.
셋째, 속도와 오버헤드 측면에서 UDP가 더 단순해서 빠르고, TCP는 상대적으로 무겁습니다.
그래서 신뢰성이 중요하면 TCP를, 실시간성이 중요하면 UDP를 사용합니다.”

이 정도면 상당히 깔끔합니다.

> 2-3. “원리 설명형” 템플릿

예: HTTP는 어떻게 동작하나요, 트랜잭션은 어떻게 보장하나요, DNS 조회 과정 설명해주세요

템플릿

목적 → 단계별 흐름 1,2,3,4 → 핵심 포인트 → 병목/주의점

말하는 흐름

“이 과정의 목적은 ~~입니다.
흐름은 보통 다음과 같습니다.
먼저 ~~하고, 그 다음 ~~하고, 마지막으로 ~~합니다.
이때 중요한 포인트는 ~~입니다.
그리고 실제 운영에서는 ~~ 문제가 생길 수 있습니다.”

예시: DNS

“DNS의 목적은 사람이 읽기 쉬운 도메인 이름을 IP 주소로 변환하는 것입니다.
흐름은 보통 브라우저 캐시 확인, OS 캐시 확인, 로컬 DNS 서버 조회, 필요 시 상위 DNS 서버들에 순차 질의하는 식으로 진행됩니다.
최종적으로 IP를 얻으면 해당 서버와 통신하게 됩니다.
이때 핵심은 캐시를 적극적으로 활용해서 조회 비용을 줄인다는 점입니다.”

> 2-4. “이유 설명형” 템플릿
예: 왜 인덱스를 쓰나요, 왜 정규화를 하나요, 왜 스레드보다 프로세스 생성 비용이 큰가요

템플릿

문제 상황 → 원인 → 해결 방식 → trade-off

말하는 흐름

“배경부터 말씀드리면, ~~한 문제가 있습니다.
그 이유는 ~~ 때문입니다.
이를 해결하기 위해 ~~를 사용합니다.
다만 이 방식에도 ~~한 비용이나 제약이 있습니다.”

예시: 인덱스

“배경부터 말씀드리면, 데이터가 많아질수록 원하는 값을 찾는 조회 비용이 커지는 문제가 있습니다.
그 이유는 매번 전체 데이터를 확인하면 탐색 범위가 너무 커지기 때문입니다.
이를 해결하기 위해 인덱스를 사용해서 특정 컬럼 기준으로 빠르게 접근합니다.
다만 인덱스가 많아지면 쓰기 성능과 저장 공간 측면에서 비용이 증가합니다.”

이렇게 말하면 단순 암기 답변보다 훨씬 좋아 보입니다.

> 2-5. “장단점형” 템플릿
> 
예: Redis 장단점, NoSQL 장단점, 멀티스레드 장단점

템플릿

한 줄 정의 → 장점 2개 → 단점 2개 → 어떤 상황에 적합한지

말하는 흐름

“A는 ~~한 기술입니다.
장점은 첫째 ~~ , 둘째 ~~입니다.
반면 단점은 ~~ , ~~가 있습니다.
그래서 ~~한 상황에서는 적합하지만, ~~한 상황에서는 주의가 필요합니다.”

> 2-6. “문제 해결형” 템플릿

예: 동시성 문제를 어떻게 해결할 건가요, DB 성능이 느릴 때 어떻게 볼 건가요

이건 실무형 면접에서 엄청 중요합니다.

템플릿

문제 정의 → 원인 후보 분리 → 확인 방법 → 해결 우선순위

말하는 흐름

“먼저 문제를 ~~로 정의할 것 같습니다.
원인은 크게 ~~ , ~~ , ~~ 세 가지로 나눠서 볼 수 있습니다.
그래서 바로 해결하기보다 먼저 로그, 메트릭, 실행 계획 같은 것으로 병목 지점을 확인하겠습니다.
그 다음 가장 영향도가 큰 부분부터 ~~ 순서로 개선하겠습니다.”

예시: DB 느릴 때

“먼저 DB가 느리다는 문제를 조회 지연인지, 락 문제인지, 커넥션 문제인지로 나눠서 볼 것 같습니다.
원인은 쿼리 자체의 비효율, 인덱스 부재, 트래픽 증가, 애플리케이션 레벨의 잘못된 호출 등으로 나눌 수 있습니다.
그래서 실행 계획과 슬로우 쿼리 로그를 먼저 확인하고, 병목 쿼리를 식별한 뒤 인덱스, 쿼리 구조, 호출 빈도 순으로 개선하겠습니다.”

이 답변은 되게 “일하는 사람”처럼 들립니다.
